深入分析：选项 B（独立模块）为什么更适合

从数据模型、制造层级、版本锁定三个维度对比

1. 数据模型差异

现有 工艺BOM (sp_bom)
bom_code — BOM编号（手动输入）
materiel_code — 物料编码
materiel_desc — 物料描述
version_number — 版本号
state — creat/pass
factory — 工厂
remark — 备注

⚠ 无 parent_id（平表）
⚠ 无 bom_type（不区分产品/工艺）
⚠ 无 locked 状态
⚠ BOM Item 功能为空白
新建 产品BOM (sp_product_bom)
bom_code — BOM编号（自动生成）
product_code — 产品物料编码
product_desc — 产品物料描述
parent_id — 父节点ID（支持树形层级）
level — 层级（0=产品/1=半成品/2=组件）
version — 版本号
status — draft/locked
remark — 备注
locked_at — 锁定时间
locked_by — 锁定人

✓ 树形结构内建支持
✓ 锁定状态独立管理

2. 制造层级结构对比

选项 A：改造现有平表
sp_bom (加 parent_id):
├─ BOM-001 台式电脑主机 (parent=null)
├─ BOM-002 台式电脑半成品 (parent=BOM-001)
├─ BOM-003 主板单元 (parent=BOM-002)
└─ BOM-004 机箱单元 (parent=BOM-002)

⚠ 所有节点平级存储，通过 parent_id 链接
⚠ 工艺BOM和产品BOM的数据混在一起
⚠ 查询/校验需要区分类型

选项 B：独立产品BOM表
sp_product_bom (父子结构):
├─ PBOM-001 台式电脑主机 [level=0, locked]
│   ├─ PBOM-002 台式电脑半成品 [level=1]
│   │   ├─ PBOM-003 主板单元 [level=2]
│   │   └─ PBOM-004 机箱单元 [level=2]
│   └─ ...
│
sp_product_bom_item (BOM行项目):
├─ PBOM-003 → CPU i7-13700 x1
├─ PBOM-003 → DDR5 32GB x2
├─ PBOM-004 → 机箱外壳 x1
└─ PBOM-004 → 电源 750W x1

✓ 数据隔离，不会与工艺BOM混淆
✓ level字段快速定位层级
✓ BOM Item 挂载在每个节点的物料清单

3. 版本锁定机制

场景 选项 A 选项 B
锁定操作 改 state='locked'，但要排除工艺BOM记录 改 status='locked'，记录 locked_at/locked_by
锁定后编辑 需在 service 层判断 state + bom 类型 controller 层统一校验 status
创建新版本 复制记录 + version+1，与其他BOM耦合 独立的新版本创建流程，清晰隔离

总结推荐

选项 B（独立模块）更适合，因为：

  1. 产品BOM 的树形层级结构与工艺BOM 的平表结构完全不同
  2. 锁定/版本管理是产品BOM 的核心需求，独立建模更清晰
  3. 现有工艺BOM 的 Item 控制器完全空白，"新建"比"改造"付出的代价差不多
  4. 数据隔离，未来两个模块各自演进，互不干扰
